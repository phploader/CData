# **1. Grundkonzepte von CData**

CData basiert auf einem einfachen, aber äußerst leistungsfähigen Prinzip:\
**Alle Daten – sowohl beim Lesen als auch beim Schreiben – werden über zwei Arrays gesteuert: `$D` und `$F`.**\
Diese beiden Strukturen bilden das Herzstück von CData und ermöglichen es, komplexe Datenmodelle ohne SQL und ohne ORM abzubilden.

---

# **$D – Der Datencontainer**

`$D` ist ein **multidimensionales Array**, das sowohl Eingabedaten (für `set_object()`) als auch Ausgabedaten (von `get_object()`) enthält.\
Die Struktur ist **immer identisch**, egal ob Daten gelesen oder geschrieben werden.

Beispiel für einen Artikel:

```php
$D['ARTICLE']['D']['id-artikel']['Active'] = 1;
$D['ARTICLE']['D']['id-artikel']['Name']   = 'Mein Artikel';
```

### Eigenschaften von `$D`:

- Jede Tabelle ist ein eigener Schlüssel (`ARTICLE`, `ATTRIBUTE`, `STOCK`, …)
- Unter `D` befinden sich die Datensätze
- Die **ID eines Datensatzes ist immer der Array-Key**
- IDs können Zahlen und Buchstaben enthalten
- CHILD‑Tabellen werden rekursiv unter `D` verschachtelt
- Die Struktur kann beliebig tief sein

Beispiel für eine tiefe Struktur:

```
ARTICLE
  D
    123
      ATTRIBUTE
        D
          Color
            LANGUAGE
              DE
                Title
```

---

# **$F – Der Filtercontainer**

`$F` wird ausschließlich für `get_object()` verwendet und bestimmt, **welche Daten geladen werden sollen**.

Beispiel: Alle Artikel laden

```php
$F['ARTICLE'] = [];
```

### WHERE‑Filter (`W`)

- `W[0]` = AND‑Block
- `W[1]` = OR‑Block

Beispiel: Aktive Artikel mit Name „Artikel A“

```php
$F['ARTICLE']['W'][0]['Active'] = 1;
$F['ARTICLE']['W'][0]['Name']   = 'Artikel A';
```

Beispiel: Zusätzlich Artikel mit Name „Artikel B“ (OR)

```php
$F['ARTICLE']['W'][1]['Name'] = 'Artikel B';
```

### Vergleichsoperatoren

```php
$F['ARTICLE']['W'][0]['Price']['>']  = 10;
$F['ARTICLE']['W'][0]['Price']['<']  = 100;
$F['ARTICLE']['W'][0]['Active']['!='] = 0;
```

### LIKE‑Varianten

```php
$F['ARTICLE']['W'][0]['Name']['LIKE']   = '%test%';
$F['ARTICLE']['W'][0]['Name']['LIKE%%'] = 'test';
$F['ARTICLE']['W'][0]['Name']['LIKE%-'] = 'test';
$F['ARTICLE']['W'][0]['Name']['LIKE-%'] = 'test';
```

### IN / NOT IN

```php
$F['ARTICLE']['W'][0]['ID'] = ['1','2','3'];          // IN
$F['ARTICLE']['W'][0]['ID']['NOTIN'] = ['1','2','3']; // NOT IN
```

---

# **ORDER & LIMIT**

Sortierung:

```php
$F['ARTICLE']['O'][0]['Name'] = 'ASC';
```

Limit + Offset:

```php
$F['ARTICLE']['L']['STEP']  = 10;
$F['ARTICLE']['L']['START'] = 0;
```

---

# **Pattern – Die Strukturdefinition**

CData erzeugt Tabellen automatisch anhand eines Pattern‑Arrays:

```php
$D['PATTERN']['ARTICLE'] = [
    'Active'   => ['Type' => 'checkbox'],
    'Name'     => ['Type' => 'text'],
    'ParentId' => ['Type' => 'id', 'ForeignKey' => 1],
];
```

CHILD‑Tabellen werden über `D` definiert:

```php
$D['PATTERN']['ARTICLE']['D']['ATTRIBUTE'] = [
    'Active' => ['Type' => 'checkbox'],
    'Name'   => ['Type' => 'text'],
];
```

### Unterstützte Typen:

| Type | Bedeutung |
|------|-----------|
| checkbox | bool (0/1) |
| text | Text |
| id | alphanumerische ID |
| number | numerischer Wert |
| json | JSON‑Wert |
| date | Datum |

---

# **get_object() – Daten lesen**

```php
$db->get_object($D, $F);
```

- `$F` bestimmt, was geladen wird
- `$D` wird mit den Ergebnissen gefüllt
- CHILD‑Tabellen werden automatisch rekursiv geladen

---

# **set_object() – Daten speichern**

```php
$db->set_object($D);
```

CData erkennt automatisch:

- Insert
- Update
- Delete

### Datensatz löschen:

```php
$D['ARTICLE']['D']['id-artikel']['__DELETE__'] = 1;
```

### CHILD‑Element löschen:

```php
$D['ARTICLE']['D']['id-artikel']['ATTRIBUTE']['Color']['__DELETE__'] = 1;
```

### Alle CHILD‑Elemente löschen:

```php
$D['ARTICLE']['D']['id-artikel']['ATTRIBUTE']['__DELETE__'] = 1;
```

---

# **Zusammenfassung**

CData basiert auf drei Grundpfeilern:

1. **$D** – Datencontainer (Lesen & Schreiben)
2. **$F** – Filtercontainer (Abfragen)
3. **PATTERN** – Strukturdefinition (Tabellen & Felder)

Mit diesen drei Elementen lassen sich beliebig komplexe Datenmodelle ohne SQL und ohne ORM abbilden.
