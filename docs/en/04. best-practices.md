# **8. Best Practices**

This chapter summarizes proven practices that help you use CData efficiently, stably, and cleanly.
These recommendations are based on typical use cases, frequent pitfalls, and optimal structuring strategies for complex data models.

---

# **8.1 Use clean and stable IDs**

CData always uses the array key as the record ID:

```php
$D['ARTICLE']['D']['A100']
```

Best practices:

- IDs should be **unique**
- IDs should **not change dynamically**
- IDs may be **alphanumeric**
- IDs should **not contain spaces**

Good examples:

```
A100
ART-001
COLOR_RED
```

Bad examples:

```
123 456
Article 1
```

---

# **8.2 Define patterns clearly and completely**

The pattern is the foundation of CData.
Be sure to:

- give every table a complete pattern
- always define CHILD tables under `D`
- choose sensible data types
- set foreign keys only when needed

Example:

```php
$D['PATTERN']['ARTICLE'] = [
    'Active' => ['Type' => 'checkbox'],
    'Name'   => ['Type' => 'text'],
];
```

---

# **8.3 Keep `$D` and `$F` strictly separate**

- `$D` = data container (reading & writing)
- `$F` = filter container (queries)

Never mix these two structures.

---

# **8.4 Load only the data you need**

CData can load very large structures.
Load only what you really need:

```php
$F['ARTICLE'] = []; // loads only articles
```

Or:

```php
$F['ARTICLE']['ATTRIBUTE'] = []; // loads articles + attributes
```

Avoid:

```php
$F['ARTICLE']['ATTRIBUTE']['PROPERTY']['LANGUAGE']['META'] = [];
```

when you only need articles.

---

# **8.5 Use CHILD filters intentionally**

Difference:

### Filter articles by attributes

```php
$F['ARTICLE']['W'][0]['ATTRIBUTE']['W'][0]['Name'] = 'Red';
```

→ only articles with attribute "Red"

### Load articles, but filter attributes

```php
$F['ARTICLE']['ATTRIBUTE']['W'][0]['Name'] = 'Red';
```

→ all articles, but only attributes named "Red"

This is a common mistake – pay attention to filter placement.

---

# **8.6 Use `__DELETE__` sparingly and deliberately**

Delete a record:

```php
$D['ARTICLE']['D']['A100']['__DELETE__'] = 1;
```

Delete a single CHILD element:

```php
$D['ARTICLE']['D']['A100']['ATTRIBUTE']['Color']['__DELETE__'] = 1;
```

Delete all CHILD elements:

```php
$D['ARTICLE']['D']['A100']['ATTRIBUTE']['__DELETE__'] = 1;
```

Be careful to:

- not set unnecessary delete markers
- place child deletions intentionally

---

# **8.7 Plan deep structures sensibly**

CData supports unlimited depth – but:

- plan the structure in advance
- avoid unnecessarily deep nesting
- use CHILD tables only when logically necessary
- keep the structure consistent

Good structure example:

```
ARTICLE
  ATTRIBUTE
    LANGUAGE
```

Bad structure example:

```
ARTICLE
  ATTRIBUTE
    PROPERTY
      META
        EXTRA
          MORE
            AGAIN
```

---

# **8.8 Combine filters, sorting, and limit**

A typical example:

```php
$F['ARTICLE']['W'][0]['Active'] = 1;
$F['ARTICLE']['O'][0]['Name']   = 'ASC';
$F['ARTICLE']['L']['STEP']      = 20;
$F['ARTICLE']['L']['START']     = 0;
```

Best practices:

- define sorting before limit
- keep filters as specific as possible
- apply CHILD sorting only when needed

---

# **8.9 Save multiple tables simultaneously**

CData can save multiple tables in one pass:

```php
$D['ARTICLE']   = [...];
$D['ATTRIBUTE'] = [...];
$D['STOCK']     = [...];

$db->set_object($D);
```

Best practices:

- structure `$D` clearly
- do not insert unnecessary tables
- always place CHILD data under `D`

---

# **8.10 Debugging Tips**

- inspect `$D` after `get_object()`
- output `$F` before loading
- verify patterns for completeness
- check IDs when data is not saved
- verify `__DELETE__` when data disappears

---

# **Summary**

- keep IDs stable and unique
- define patterns cleanly
- keep `$D` and `$F` separate
- load only required data
- use CHILD filters correctly
- use delete markers deliberately
- plan deep structures carefully
- combine filters, sorting, and limit
- save multiple tables clearly
- review `$D` and `$F` regularly

These best practices help you use CData efficiently, stable, and error-free.
